# Phase 5 — Claude Haiku AI 캐릭터

**목표**: Claude Haiku 기반 AI 캐릭터가 방에 상시 존재하며, 대화 맥락을 관찰해 자율적으로 발화한다. AI의 메시지는 인간과 완전히 동일한 파이프라인(감정 인식→포즈 매칭→패널 배치)을 통과해야 한다.

**데모 기준**: 방에 인간 참여자들이 대화하는 중 AI 캐릭터가 맥락에 맞게 끼어들어 발언하고, 다른 캐릭터와 똑같은 방식(포즈/패널 배치/말풍선)으로 만화에 등장한다. 이름을 부르면 반응 확률이 높아지고, 스팸처럼 매 메시지마다 끼어들지 않는다.

## 선행 조건
- Phase 4 완료 (전체 프로토콜, 서버가 canonical 소스로 확립됨)

## 작업 목록

### 1. AI 액터 등록
- [ ] AI 캐릭터를 일반 room member로 등록 (`actor_id`, 전용 아바타/페르소나 선택)
- [ ] 실제 WS 연결 없이 서버 내부에서 인간과 동일한 `handleSay(roomId, actorId, text, mode)` 진입점을 직접 호출하는 구조 확인 (Phase 1~4에서 이미 이 함수가 단일 진입점이어야 함 — 아니라면 리팩터링)

### 2. 발화 트리거 휴리스틱 (`apps/server/src/ai/trigger.ts`)
- [ ] 쿨다운 타이머 (최소 30~60초, 설정 가능)
- [ ] 호명 감지: `comic-engine`의 `checkWord` 매처를 재사용해 AI 캐릭터 이름/닉네임 언급 감지 → 트리거 확률/우선순위 상승
- [ ] 무활동 감쇠: 대화가 AI를 향하지 않고 계속되면 트리거 확률 감소
- [ ] 낮은 확률의 "잡담" 트리거 (침묵이 길어질 때 가끔)
- [ ] 트리거 로직 단위 테스트 (다양한 대화 시나리오로 스팸 발생 여부 검증)

### 3. Haiku 연동 (`apps/server/src/ai/aiCharacter.ts`)
- [ ] Anthropic TypeScript SDK 설치, API 키는 환경변수(`ANTHROPIC_API_KEY`)로 관리 (레포에 커밋 금지 — `.env` + `.gitignore` 확인)
- [ ] 페르소나 시스템 프롬프트 설계 (이름, 성격, 말투, 방 맥락 설명)
- [ ] 트리거 시 최근 N개 `HistoryEntry`를 프롬프트로 포맷 → `claude-haiku-4-5` 호출 (non-streaming, 짧은 `max_tokens`)
- [ ] 응답 텍스트를 `handleSay()`로 전달 → 정상적으로 규칙엔진/포즈매칭/레이아웃을 통과하는지 확인
- [ ] 오류/레이트리밋 시 조용히 스킵 (재시도 스팸 금지, 로그만 남김)

### 4. 운영 안전장치
- [ ] 요청 빈도 상한(rate limit) — 실수로 트리거가 폭주해도 API 비용 폭탄 방지
- [ ] AI 캐릭터 on/off 토글 (방장 또는 서버 설정)
- [ ] 컨텐츠 필터링 여부 검토 (Haiku 자체 안전장치에 의존, 필요시 시스템 프롬프트에 room 톤 가이드 추가)

## 완료 조건 (Acceptance)
- [ ] AI 캐릭터가 자연스러운 빈도로(스팸 아님) 대화에 참여
- [ ] 이름을 부르면 응답 확률이 눈에 띄게 높아짐
- [ ] AI 메시지가 다른 사용자 메시지와 렌더링 파이프라인상 구분 불가능함(같은 감정엔진/포즈매칭/패널배치)
- [ ] API 키가 코드/커밋에 노출되지 않음
